#  AI Agent

## 1.  AI에이전트의 개념 및 정의
- 에이전트: 특정 목적을 위해 자율적으로 행동하는 개체
- 챗봇(Chat-bot): 대화(Chat)와 로봇(Bot)을 합친 용어로 사용자와 텍스트나 음성으로 상호작용하는 대화 인터페이스에 초점을 맞춘 도구
- AI에이전트: 목표를 달성하기 위해 주변 환경을 관찰하고 보유한 도구를 활용하여 행동하는 자율적인 애플리케이션

### 기존 챗봇 vs AI에이전트 비교
|구분|기존 챗봇|AI 에이전트|
|----|----|----|
|작업 방식|일회성 질답|지속적 작업 수행|
|대응 능력|수동적 응답|능동적 문제 해결|
|도구 활용|단일 기능|다중 도구 활용|
|계획 수립|불가능|자율적 계획 수립|

### ReAct(Reasoning and Acting) 패턴
- AI에이전트는 ReAct 패턴을 기반으로 동작, 추론(Reasoning)과 행동(Acting)을 번갈아 수행하며 문제를 해결하는 방식으로 에이전트의 핵심 작동 원리
#### ReAct 패턴의 다섯 단계
1. 사고(Thought): 문제를 분석하고, 사용할 도구 결정
2. 행동(Action): 선택한 도구를 실제로 실행
3. 관찰(Observation): 도구 실행 결과를 분석하고 다음 단계 결정
4. 반복(Loop): 목표에 도달할 때까지 사고 -> 행동 -> 관찰 과정 반복
5. 최종 답변(Final Answer): 수집한 모든 정보를 종합해 최종 결과 생성

### 효과적인 에이전트 프롬프트 작성법
- 실제 에이전트 프롬프트는 다음과 같은 구조로 작성
> 당신은 질문에 답하는 도우미입니다. 당신의 임무는 제공된 자료(context)를 바탕으로 사용자의 질문에 답하는 것입니다.
>
> 다음 지침을 순서대로 따릅니다:
> 1. 먼저 "pdf_search" 도구를 사용해 PDF 문서에서 정보를 찾습니다.
> 2. 그런 다음 "relevance_check" 도구를 사용해, 찾은 정보가 사용자의 질문과 얼마나 관련있는지 확인합니다.
> 3. 만약 "relevance_check" 결과가 심각(serious)하다고 나오면, 새로운 검색어를 만들어 다시 "pdf_search" 도구로 검색합니다.
> 4. 답변에는 반드시 "relevance_check" 도구 호출 결과를 포합해야 합니다.
>
> 기억하세요:
> - "ㄱrelevance_check" 결과가 심각하면, 새로운 검색어로 다시 "pdf_search"를 실행해야 합니다.
> - 이 과정을 최대 20번까지 반복할 수 있습니다.

#### 에이전트 프로므프트 작성 시 주의사항
- 에이전트 프롬프트에서는 도구 이름을 정확히 명시하세요. "검색해라."가 아니라 "tavily_search 도구를 사용해라." 처럼 구체적으로 써야 에이전트가 올바른 도구를 선택할 수 있습니다.

### 에이전트 루프 (Agent Loop)의 작동 원리

1. 입력 및 상황 설정 (Input & Context):  
에이전트 시스템이 사용자 요청과 현재 상태(과거 대화 내역, 가용한 툴 목록 등)를 정리해 LLM에게 전달합니다.

2. 판단 및 추론 (Reasoning):  
LLM은 전달받은 상황을 보고 "지금은 직접 답할 수 없으니 검색 툴을 써야겠다"라고 판단합니다.

3. 도구 호출 (Tool Call / Action):  
LLM이 직접 툴을 실행하는 게 아니라, **"툴 이름: Search, 입력값: 서울 날씨"**라는 메시지를 내뱉습니다.

4. 실행 (Execution):  
에이전트 시스템이 이 메시지를 읽고, 실제로 구글 API나 브라우저를 구동해 결과를 가져옵니다.

5. 관찰 및 피드백 (Observation):  
시스템이 가져온 결과(예: "서울은 맑음, 20도")를 다시 LLM에게 **"툴 실행 결과는 다음과 같아. 이제 어떻게 할래?"**라고 물어봅니다.

6. 반복 및 완료:  
LLM이 정보를 충분히 얻었다고 판단하면 최종 답변을 생성하고 루프를 종료합니다.

#### 에이전트 환경: 에이전트 시스템이 구동되는 전체 인프라
- 메모리: LLM은 기본적으로 단기 기억상실증 환자와 같다. 에이전트 환경은 이전 단계에서 어떤 툴을 썻고 어떤 결과가 나왔는지 기록해 두었다가 다음 단계의 LLM에게 계속 알려준다.
- 제어 로직: 루프를 몇 번 돌릴지, 에러가 나면 어떻게 재시도할지, 최종 답변은 어떤 형식으로 출력할지 관리하는 '관리 코드'이다.
- 보안 샌드박스: LLM이 생성한 파이썬 코드가 내 컴퓨터를 망가뜨리지 않도록, 격리된 가상 공간에서 코드를 실행하는 안전한 환경을 제공한다.

#### 에이전트 시스템(프레임워크/플랫폼)
- LangChain
- OpenClaw
- Dify

#### 프레임워크(Framework): 뼈대와 규칙
: 프레임워크는 소프트웨어를 개발할 때 기초가 되는 **구조(뼈대)**와 규칙을 제공하는 환경이다.
- 개념: 개발자가 밑바닥부터 모든 코드를 짜지 않도록, 자주 쓰이는 기능들을 미리 만들어둔 '반제품'상태의 도구 모음이다.
  (단, 프레임워크가 정해둔 규칙과 흐름안에서만 개발해야 한다.)
- 특징(제어의 역전): 보통은 개발자가 짠 코드가 도구를 불러오지만, 프레임워크는 프레임워크가 개발자의 코드를 불러와서 실행한다.
- 비유: **조립식 건축 뼈대**이다. 철골 구조와 배관 위치는 이미 정해져 있다. 개발자는 그 뼈대 위에 어떤 벽돌을 쌓고 무슨 색 페인트를 칠할지(비즈니스 로직)만 고민하면 된다.
- 예시: LangCahin, 에이전트를 만들기 위한 부품(메모리, 툴, 프롬프트 템플릿)은 다 제공하지만, 이것들을 엮어서 프로그램으로 완성하고 서버에 올리는 건 전적으로 개발자가 파이썬 코드로 직접 해야한다.

#### 플랫폼(Platform): 완성된 무대와 인프라
: 플랫폼은 애플리케이션을 개발하는 것을 넘어, 그것을 **실행하고, 서비스하고, 관리할 수 있는 종합적인 환경(기반)**을 의미한다.
- 개념: 코딩을 넘어 소프트웨어가 실제로 구동되는 토대이다. 운영체제, 하드웨어, 네트워크 그리고 프레임워크까지 모두 포함하는 거대한 개념이다.
- 특징: 모든 인프라가 갖춰져 있어 사용자는 '콘텐츠(또는 서비스)'를 올리기만 하면 된다.
- 비유: **입점 준비가 끝난 대형 백화점**이다. 전기, 보안, 결제 시스템, 인테리어까지 다 되어있다. 상인은 물건(아이디어)만 가져와서 매대에 올려놓고 바로 장사를 시작하면 된다.
- 예시: Dify, 코드를 짤 필요 없이 웹 브라우저에 접속햇 ㅓ마우스 클릭 몇 번으로 에이전트를 만들 수 있다. 만들어진 에이전트를 구동할 서버, 대화 내역을 저장할 데이터베이스, 남들에게 공유할 웹 링크까지 플랫폼에서 다 알아서 제공한다.
